# Протокол сессий проекта Drape

Правила игры для всех чатов в этом Project. Любой Claude, читающий
Project Knowledge, обязан следовать этому документу.

## Контекст

Проектируется веб-приложение Drape для визуализации пиксель-арт
мозаики на стенах. Этап планирования (6 сессий) завершён, дальше —
пост-аудиты, проработка открытых вопросов и подготовка к реализации.

Все документы планирования живут в репозитории:
https://github.com/vityaayaa/drape/tree/main/planning

Project Knowledge подключён к этой папке через GitHub Integration.
**После каждого handoff** пользователь применяет правки в репозитории
и нажимает **Sync в Knowledge** перед открытием нового чата —
иначе следующий чат прочитает устаревшие файлы.

## Файлы в /planning/

### Постоянные

- `HANDOFF_PROTOCOL.md` — этот файл, правила работы
- `PLAN.md` — карта сессий и их статус
- `PROJECT.md` — суть приложения (заморожен)
- `DECISIONS.md` — журнал всех значимых решений
- `OPEN_QUESTIONS.md` — список открытых вопросов

### Тематические

- `ROOM_MODEL.md` — модель комнаты, стен, плиток, масок
- `PIXELIZATION.md` — обработка изображения, палитра, превью
- `VIEWER_3D.md` — 3D-вьюер, камера, освещение, рендер
- `EXPORT.md` — сохранение проекта, схема, спецификация, режим укладки
- `TECH_STACK.md` — технологии, структура папок, этапы реализации

## Триггеры пользователя

| Триггер | Что делает Claude |
|---|---|
| `СЕССИЯ N: [НАЗВАНИЕ]` | Запускает протокол открытия плановой сессии |
| `ПОСТ-АУДИТ` или `АУДИТ` | Запускает протокол открытия пост-аудита |
| `ЗАКРЫВАЕМ. Дай handoff.` | Выдаёт пакет закрытия |
| `ВРЕМЕННЫЙ HANDOFF` | Выдаёт промежуточный пакет (без закрытия) |

## Протокол открытия

При получении триггера Claude:

1. Читает все файлы из /planning/, перечисленные в стартовом
   сообщении (или все, если не уточнено).
2. В первом ответе делает **verification check**:
   - Что прочитал
   - 3–5 ключевых точек напряжения, которые видит
   - Предлагаемый план обсуждения
   - Спрашивает: «План ОК или скорректировать?»
3. Не лезет в правки до подтверждения плана.

Если пользователь говорит «контекст не тот» — переоткрыть чат
после Sync в Knowledge.

## Поведение в течение сессии

- Идти по согласованному плану, по одной теме за раз.
- Каждое значимое решение проговаривать явно:
  «Решение: X. Альтернатива: Y. Почему X: Z.»
- Максимум 2–3 альтернативы при выборе.
- Если возникает вопрос вне плана — добавлять в OPEN_QUESTIONS,
  не уводить разговор.
- Не выдавать изменения файлов посреди сессии — только в handoff.
- Не повторять содержимое Knowledge своими словами без запроса.
- Объяснять технические термины простым языком — пользователь
  не программист.
- Обращение на «ты».

## Протокол закрытия

При получении триггера `ЗАКРЫВАЕМ. Дай handoff.` Claude выдаёт
строго два блока в таком порядке:

### Блок А — Изменения в файлах

Используются четыре операции:

```
=== ДОПОЛНИТЬ: planning/имя.md ===
В конец файла добавить:
[блок текста]
=== КОНЕЦ ===
```

```
=== ЗАМЕНИТЬ В: planning/имя.md ===
Найти:
[точный фрагмент как в файле, со всеми пустыми строками]
Заменить на:
[новый фрагмент]
=== КОНЕЦ ===
```

```
=== СОЗДАТЬ: planning/имя.md ===
[полное содержимое нового файла]
=== КОНЕЦ ===
```

```
=== ПЕРЕЗАПИСАТЬ: planning/имя.md ===
Перед вставкой удали старое содержимое.
[полное новое содержимое]
=== КОНЕЦ ===
```

### Когда какую операцию использовать

- **ДОПОЛНИТЬ** — для DECISIONS.md и OPEN_QUESTIONS.md
  (новые записи в конец, старое не трогать).
- **ЗАМЕНИТЬ** — для точечных правок (статус решения, статус
  пункта в PLAN.md, обновление одного абзаца).
- **СОЗДАТЬ** — только для файлов, которых ещё нет.
- **ПЕРЕЗАПИСАТЬ** — для тематических файлов после серьёзных
  правок, когда точечных ЗАМЕНИТЬ было бы 5+. Разрешено всегда
  для ROOM_MODEL, PIXELIZATION, VIEWER_3D, EXPORT, TECH_STACK.
  **Запрещено** для PROJECT.md и HANDOFF_PROTOCOL.md без явного
  разрешения пользователя в текущем чате.

### Правила ЗАМЕНИТЬ

- Фрагмент в «Найти» должен быть **уникальным в файле** и
  содержать **все строки** до естественной границы (следующий
  заголовок, пустая строка перед следующим разделом).
- Если границу непонятно где провести — использовать ПЕРЕЗАПИСАТЬ.
- Урок прошлого: неполная замена оставляет хвост старого блока
  и создаёт противоречия в файле.

### Обязательные обновления каждого закрытия

- `DECISIONS.md` — все значимые решения сессии (новые записи,
  обновление статусов отменённых/уточнённых).
- `OPEN_QUESTIONS.md` — закрыть решённые, добавить новые.
- `PLAN.md` — статус, если завершилась плановая сессия.

### Блок Г — Открытые вопросы (сводка)

Краткая таблица или список:
- какие вопросы остались открытыми после сессии
- когда их решать (фаза/этап/«пользователю самостоятельно»)

Блок Б (стартовое сообщение для следующей сессии) выдаётся
**только по явному запросу пользователя** — после планового
этапа в нём смысла нет.

## Правила обёртки код-блоков

Маркеры операций (`=== ДОПОЛНИТЬ ===`, `=== ЗАМЕНИТЬ ===` и т.д.)
пишутся как обычный текст в чате, **снаружи** код-блоков.

Содержимое для вставки в файл оборачивается в **один код-блок**
сразу после маркера операции. Внутри ровно тот текст, который
должен попасть в файл.

**Простое правило по бэктикам:**

- **По умолчанию — 4 бэктика снаружи.** Этого хватает для
  большинства .md-файлов с примерами кода внутри.
- **5 бэктиков** — только если внутри файла встречаются 4 подряд
  (редкий случай).
- **3 бэктика** — допустимо только для коротких блоков без
  примеров кода внутри.

Перед отправкой Claude мысленно проверяет: самая длинная
последовательность бэктиков внутри < количество бэктиков снаружи.
Если сомневается — берёт больше.

**Тег языка** после открывающих бэктиков:
- `markdown` для .md-файлов
- `javascript`, `typescript`, `python` и т.д. для кода
- `text` если ничего не подходит

## Правила редактирования отдельных файлов

| Файл | Что разрешено |
|---|---|
| `HANDOFF_PROTOCOL.md` | Только с явным разрешением пользователя |
| `PROJECT.md` | Только с явным разрешением пользователя |
| `PLAN.md` | Точечные ЗАМЕНИТЬ статусов, структуру не трогать |
| `DECISIONS.md` | Только ДОПОЛНИТЬ + ЗАМЕНИТЬ статусов |
| `OPEN_QUESTIONS.md` | Только ДОПОЛНИТЬ + ЗАМЕНИТЬ статусов |
| `ROOM_MODEL.md` и др. тематические | Любые операции, включая ПЕРЕЗАПИСАТЬ |

Старые записи в DECISIONS и OPEN_QUESTIONS не удалять никогда.
Если решение отменено — статус меняется, запись остаётся как след
истории.

## Стиль работы

- На «ты», простым языком, без воды.
- Технические термины — объяснять на ходу.
- Решения — формат «что → почему → альтернативы».
- Максимум 2–3 альтернативы при выборе.
- Короткий ответ — коротко. Длинный — только когда задача требует.
- Не льстить. Не хвалить без причины. Главное — найти проблемы
  и решить их.
- Если пользователь говорит «это закрыто, идём дальше» —
  не пересматривать, идти дальше.

## Чек-лист после применения handoff

Пользователь после получения пакета:
1. Применяет все блоки А в репозитории по порядку.
2. Коммитит изменения.
3. **Жмёт Sync в Knowledge.**
4. Открывает новый чат со стартовым сообщением.
