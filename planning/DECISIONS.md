## Решения

### 1. Форма комнаты задаётся вручную, не рисованием

- Сессия: 0 (зафиксировано до старта)
- Что решили: пользователь вводит количество стен и их размеры 
  числами, не рисует план комнаты сверху.
- Почему: проще в реализации, достаточно для текущих задач.
- Альтернативы: рисование плана сверху (отвергнуто как переусложнение 
  на старте, можно вернуться позже).
- Статус: активно

### 2. Размер и форма плитки — произвольные

- Сессия: 0
- Что решили: пользователь задаёт размер плитки сам, плитки могут 
  быть плоскими (мозаика) или объёмными (кирпич).
- Почему: разные сценарии использования, не привязываемся к одному бренду.
- Альтернативы: фиксированная сетка под Lego/Pixlr (отвергнуто 
  для гибкости).
- Статус: активно

### 3. Стек выбираем в Сессии 6, а не сейчас

- Сессия: 0
- Что решили: до Сессии 6 не привязываемся к конкретным библиотекам.
- Почему: сначала понять что строим, потом выбирать инструмент.
- Альтернативы: зафиксировать Three.js + React сразу (отвергнуто).
- Статус: активно

### 4. Приложение для одного пользователя, без бэкенда

- Сессия: 1
- Что решили: никаких аккаунтов, БД и серверной логики.
  Сохранение — экспорт/импорт JSON-файла проекта.
- Почему: избыточно для текущих задач, усложняет реализацию.
- Альтернативы: облачное хранение с аккаунтами (отвергнуто).
- Статус: активно

### 5. Объекты в комнате — два режима: препятствие и декор

- Сессия: 1
- Что решили: у каждого объекта переключатель режима.
  Препятствие вырезает площадь стены, декор — только визуал.
- Почему: разные сценарии использования (лестница = препятствие,
  условный куб для ориентира = декор).
- Альтернативы: только один режим (отвергнуто как негибкое).
- Статус: пересмотрено решением №56 (режим декора убран из v1)

### 6. Обработка изображений — Web Worker + OffscreenCanvas

- Сессия: 1
- Что решили: тяжёлая пикселизация выносится в фоновый поток,
  чтобы не замораживать интерфейс на слабых устройствах.
- Почему: стандартная практика для браузерных приложений с
  тяжёлой графикой, не требует сервера.
- Альтернативы: серверная обработка (отвергнуто — не хотим бэкенд).
- Статус: активно

### 7. Палитра цветов — произвольная

- Сессия: 1
- Что решили: цвета берутся напрямую из фото, без привязки
  к каталогу магазина.
- Почему: гибкость. При необходимости пользователь сам
  сопоставит цвета с реальными плитками через ИИ.
- Альтернативы: фиксированная палитра из каталога (отвергнуто).
- Статус: активно

### 8. Углы между стенами — 90° в первой версии, поле заложено

- Сессия: 2
- Что решили: в первой версии поддерживаем только прямые углы.
  Поле `angle` добавлено в модель стены для будущего расширения.
- Почему: произвольные углы усложняют 3D и расчёт сетки.
  Покрывает 95% реальных комнат.
- Альтернативы: произвольный угол сразу (отвергнуто как преждевременное).
- Статус: активно

### 9. Плитка — глобальные параметры с переопределением на стене

- Сессия: 2
- Что решили: параметры плитки задаются один раз глобально,
  на уровне стены можно переопределить.
- Почему: гибкость без усложнения интерфейса.
- Альтернативы: только глобально (отвергнуто как негибкое).
- Статус: уточнено решением №38 (поле tile_thickness убрано из стены, остаётся только у плитки)

### 10. Угловой конфликт объёмных плиток — одна стена перекрывает

- Сессия: 2
- Что решили: на стыке чётная стена перекрывает нечётную.
  Пользователь может настроить направление для каждого угла.
- Почему: проще в расчёте и рендере, соответствует реальной практике.
- Альтернативы: угловая плитка, скос 45° (отвергнуты как сложные).
- Статус: уточнено решениями №46 (формула сетки) и №47 (правило
  «толще перекрывает тоньше; при равной — меньший номер»)

### 11. Остаток сетки — выбор пользователя между обрезкой и растяжкой шва

- Сессия: 2
- Что решили: если стена не делится нацело на плитки — два режима:
  обрезка крайних плиток или равномерная растяжка швов.
- Почему: разные предпочтения у разных пользователей.
- Альтернативы: центрирование (отвергнуто пользователем).
- Статус: пересмотрено решением №55 (растяжка швов убрана,
  остаётся только обрезка крайней плитки)

### 12. Форматы загружаемых изображений — JPEG, PNG, WebP

- Сессия: 3
- Что решили: поддерживаем три формата. Мягкий лимит ~20 МБ с предупреждением, не блокировать.
- Почему: все три читаются браузером нативно, без библиотек. GIF и SVG не нужны.
- Альтернативы: расширенный список форматов (отвергнуто как избыточное).
- Статус: активно

### 13. Алгоритм пикселизации — усреднение (area averaging)

- Сессия: 3
- Что решили: цвет каждой плитки = средний цвет всех пикселей оригинала в её области.
- Почему: точнее передаёт реальный цвет фото, особенно при делении одного изображения на несколько стен.
- Альтернативы: nearest-neighbor (отвергнуто — случайный пиксель даёт неточный цвет).
- Статус: активно

### 14. Палитра цветов — полная по умолчанию, квантизация опционально

- Сессия: 3
- Что решили: по умолчанию все уникальные цвета после пикселизации. Квантизация включается отдельно, пользователь задаёт максимум цветов (алгоритм — медианное сечение). Редактирование палитры доступно всегда.
- Почему: сценарии разные — плитки, краска, печать. Не ограничиваем заранее.
- Альтернативы: фиксированная квантизация по умолчанию (отвергнуто — слишком жёстко).
- Статус: пересмотрено решениями №39 (квантизация обязательна,
  дефолт 32 цвета) и №49 (палитра read-only, ручное редактирование убрано)

### 15. Распределение изображения по стенам — настраиваемая панорама

- Сессия: 3
- Что решили: одно фото растягивается на выбранные пользователем стены пропорционально их полезной ширине (за вычетом объектов-препятствий). Стены можно переименовывать. Каждой стене можно назначить своё фото или оставить пустой.
- Почему: объекты-препятствия вырезают площадь, натяжка должна учитывать полезную, а не полную ширину стены.
- Альтернативы: фиксированная панорама на все стены (отвергнуто — негибко).
- Статус: уточнено решением №66 (распределение пропорционально
  сеточной ширине стен; маски в распределении не участвуют —
  фото идёт цельным полотном поверх развёртки, кусок над маской
  просто не отрисовывается)

### 16. Обрезка изображения при несовпадении пропорций

- Сессия: 3
- Что решили: по умолчанию обрезка (crop) с центрированием и возможностью сдвига кадра. Переключатель на fit (вписать целиком с полосами).
- Почему: обрезка не искажает пропорции, выглядит естественнее для мозаики.
- Альтернативы: только fit (отвергнуто — искажения пропорций неестественны).
- Статус: активно

### 17. Превью результата — два уровня

- Сессия: 3
- Что решили: плоское 2D-превью (развёрнутые стены) обновляется в реальном времени через Web Worker; 3D — финальный просмотр в сессии 4.
- Почему: плоское превью быстрее и удобнее для настройки палитры и параметров пикселизации.
- Альтернативы: только 3D-превью (отвергнуто — тяжело и медленно для итеративной настройки).
- Статус: уточнено решением №44 (двухэтапное превью с явным переходом)

### 18. Тип проекции — перспектива с переключателем на изометрию

- Сессия: 4
- Что решили: по умолчанию перспектива, переключатель на изометрию доступен в настройках.
- Почему: перспектива естественнее для оценки реального вида комнаты.
- Альтернативы: только изометрия (отвергнуто — выглядит как схема).
- Статус: активно

### 19. Режимы камеры — орбита и внутри

- Сессия: 4
- Что решили: два режима. Орбита — камера вращается вокруг центра снаружи.
  Внутри — камера в центре комнаты, крутит головой. Перемещение по комнате — вне скоупа v1.
- Почему: мобильный приоритет, WASD не работает на телефоне.
- Альтернативы: свободная камера с перемещением (отложено на будущее).
- Статус: пересмотрено решением №41 (режим «внутри» убран из v1, остаётся только орбита)

### 20. Освещение — два режима без пользовательских настроек

- Сессия: 4
- Что решили: flat (равномерный) и объёмный (ambient + направленный сверху, без теней).
  Переключаемо. Интенсивность фиксированная.
- Почему: баланс между визуальным качеством и точностью цвета.
- Альтернативы: динамические тени (отвергнуто — тяжело для мобильных).
- Статус: пересмотрено решением №40 (только flat в v1)

### 21. Объёмные плитки в 3D — блок при толщине > 0

- Сессия: 4
- Что решили: если толщина > 0 — плитка рендерится как объёмный блок.
  Если толщина = 0 — стена плоская. Геометрия упрощённая.
- Почему: соответствует реальной физике укладки, реализуемо без фотореализма.
- Альтернативы: всегда плоские с имитацией объёма (отвергнуто).
- Статус: активно

### 22. Состояния вьюера — три: loading, пустое, ошибка

- Сессия: 4
- Что решили: все три реализуем. Пустое состояние не блокирует — комната
  видна сразу с серыми стенами.
- Почему: без состояний интерфейс выглядит сломанным при задержках или ошибках.
- Альтернативы: только loading (отвергнуто как недостаточно).
- Статус: активно

### 23. AR — вне скоупа

- Сессия: 4
- Что решили: AR не включаем в v1.
- Почему: WebXR плохо работает на iOS Safari, усложняет разработку.
- Альтернативы: включить как эксперимент (отвергнуто).
- Статус: активно

### 24. Сохранение проекта — JSON без фотографий

- Сессия: 5
- Что решили: JSON не включает фотографии. Пользователь хранит их сам
  и прикрепляет заново при загрузке проекта.
- Почему: фото могут весить десятки МБ, JSON остаётся лёгким.
- Альтернативы: ZIP-архив с фото внутри (отложено как усложнение).
- Статус: отменено решением №35 (фото включаются в JSON через base64)

### 25. Раздел схемы и спецификации — внутри приложения, без отдельного файла для просмотра

- Сессия: 5
- Что решили: схема и спецификация живут в отдельном разделе приложения,
  не экспортируются для просмотра — только для хранения/передачи (SVG).
- Почему: не нужно открывать внешние файлы, вся интерактивность внутри.
- Альтернативы: HTML-файл для скачивания (отвергнуто — лишний шаг).
- Статус: активно

### 26. Компоновка раздела — split-screen с перетаскиваемой границей

- Сессия: 5
- Что решили: верхняя часть — развёртка схемы (зумируемая),
  нижняя — легенда и спецификация (прокручиваемая).
  Граница перетаскивается.
- Почему: схема горизонтальная и невысокая, снизу остаётся место
  для спецификации. Обе части нужны одновременно.
- Альтернативы: вкладки (отвергнуто — неудобно переключаться).
- Статус: активно

### 27. Разделители стен на развёртке — зазор с названием

- Сессия: 5
- Что решили: между стенами зазор (не линия), название стены
  размещается в зазоре и не накладывается на плитки.
- Почему: линия может слиться со швом плитки, зазор — нет.
- Альтернативы: линия-разделитель (отвергнуто).
- Статус: активно

### 28. Экспорт итогового результата — SVG

- Сессия: 5
- Что решили: один SVG-файл содержит развёртку, легенду и спецификацию.
- Почему: бесконечное масштабирование, малый вес, открывается
  в браузере без программ. PNG теряет чёткость при зуме.
- Альтернативы: PNG (отвергнуто — потеря качества при зуме),
  PDF (отвергнуто — избыточно).
- Статус: активно

### 29. Веб-фреймворк — React

- Сессия: 6
- Что решили: React как основной UI-фреймворк.
- Почему: экосистема вокруг 3D (React Three Fiber), хорошая поддержка в Claude Code.
- Альтернативы: Vue 3 (слабее экосистема 3D), Vanilla JS (много ручной работы).
- Статус: активно

### 30. 3D-библиотека — React Three Fiber + Three.js

- Сессия: 6
- Что решили: R3F как обёртка над Three.js. Дополнительно @react-three/drei.
- Почему: чистый код в React-стиле, готовые OrbitControls, отличная поддержка в Claude Code.
- Альтернативы: Babylon.js (избыточен для наших задач).
- Статус: активно

### 31. Обработка изображений — свой Web Worker без библиотек

- Сессия: 6
- Что решили: area averaging и квантизация пишутся вручную на чистом JS.
- Почему: нужен доступ к цвету каждой плитки — готовые библиотеки не закрывают это чисто.
- Альтернативы: Pica (только ресайз, не даёт цвет ячеек), Jimp/Sharp (для Node.js, не браузер).
- Статус: активно

### 32. Хранение данных — React state + автосохранение в localStorage

- Сессия: 6
- Что решили: состояние в React, при каждом изменении дублируется в localStorage.
- Почему: защита от случайной потери при закрытии вкладки. Лимит 5 МБ не проблема — фото не храним.
- Альтернативы: только React state (риск потери работы).
- Статус: отменено решением №34 (IndexedDB вместо localStorage)

### 33. Объекты в комнате — прямоугольные маски на стенах

- Сессия: пост-аудит
- Что решили: объект на стене — это прямоугольная маска (поля: x, y, width, height, mode, name). Препятствие исключает попавшие в маску плитки из расчёта и спецификации; декор показывается в 3D как полупрозрачный блок и на расчёт не влияет. Сложные формы (П-образная лестница) собираются из нескольких прямоугольных масок.
- Почему: 3D-модели объектов — большая работа без пропорциональной отдачи. Маски решают единственную нужную задачу — корректный расчёт плиток с учётом перекрытий.
- Альтернативы: 3D-модели с пересечениями (отвергнуто — переусложнение); только визуальная пометка без вычета (отвергнуто — не даёт точную спецификацию).
- Статус: активно

### 34. Локальное хранение — IndexedDB

- Сессия: пост-аудит
- Что решили: всё рабочее состояние (настройки, фото, кэш пикселизации, история undo/redo) автосохраняется в IndexedDB браузера.
- Почему: IndexedDB встроена в браузер, не требует сервера и внешних сервисов. localStorage упирается в 5 МБ и не хранит бинарные данные — IndexedDB снимает оба ограничения.
- Альтернативы: localStorage (отвергнуто — лимит и нет бинарных данных); сервер (отвергнуто — нет бэкенда).
- Статус: активно. Отменяет решение №32.

### 35. Экспорт проекта — единый JSON с фото в base64

- Сессия: пост-аудит
- Что решили: при экспорте скачивается один JSON-файл, фотографии включены внутрь как base64. Импорт — выбор одного файла, всё восстанавливается.
- Почему: один файл проще для пользователя, чем JSON+фото отдельно. Размер 5–50 МБ приемлем для редкой операции бэкапа.
- Альтернативы: JSON и фото отдельно (отвергнуто — пользователь жонглирует файлами); ZIP-архив (отвергнуто — требует библиотеку, воспринимается как папка).
- Статус: активно. Отменяет решение №24.

### 36. Затирка — независима от толщины плитки

- Сессия: пост-аудит
- Что решили: ширина шва (grout_width) и толщина плитки (tile_thickness) — независимые параметры. Шов виден всегда, когда grout_width > 0, в том числе для плоских плиток.
- Почему: пользователь может хотеть зазор между плоскими элементами (бумажная мозаика, раскраска с разделительными линиями). Жёсткая связка лишает гибкости.
- Альтернативы: затирка только при объёмных плитках (отвергнуто).
- Статус: активно

### 37. Единицы измерения — см для стен, мм для плиток

- Сессия: пост-аудит
- Что решили: размеры стен и масок — в сантиметрах. Размеры плиток и швов — в миллиметрах. Каждое поле интерфейса подписано единицей.

### 39. Палитра — квантизация обязательна, дефолт 32 цвета

### 45. Режим укладки — пошаговый, два режима обхода

Два режима обхода: «по рядам» (слева-направо снизу-вверх) и «по цветам» (все плитки одного цвета подряд). Тап на любую плитку в превью — переход к ней.

### 46. Толщина стены не моделируется, в формуле сетки участвует tile_thickness соседних стен

- Сессия: пост-аудит 2
- Что решили: понятия «толщина стены» в модели нет. В формуле полезной ширины участвует tile_thickness соседних стен, перекрывающих текущую в углу.
  Корректная формула:
  `полезная_ширина = length − tile_thickness_левой_соседней_стены (если она перекрывает) − tile_thickness_правой_соседней_стены (если она перекрывает)`
- Почему: формулировка «толщина перекрывающих стен» в первом аудите была неточной — стена сама не имеет толщины в нашей модели, мы рисуем только внутреннюю поверхность. Имелась в виду толщина плитки соседней стены.
- Альтернативы: добавить поле толщины стены (отвергнуто — не используется ни в одном расчёте и не отображается в 3D).
- Статус: активно. Уточняет решение №10.

### 47. При смешанной толщине плиток на углу — толще перекрывает тоньше; при равной — стена с меньшим номером

- Сессия: пост-аудит 2
- Что решили: дефолтное правило перекрытия в углу — стена с более толстой плиткой перекрывает соседнюю с более тонкой. При равной толщине перекрывает стена с меньшим порядковым номером. Пользователь может переопределить вручную для каждого угла.
- Почему: визуально логичнее — тонкая плитка прижимается к торцу толстой. При равной толщине нужно детерминированное правило, чтобы поведение было предсказуемым.
- Альтернативы: жёсткое «чётная перекрывает нечётную» (отвергнуто — игнорирует физику стыка).
- Статус: активно. Уточняет решения №10 и №46.

### 48. Палитра — одна на проект, бюджет цветов настраивается

### 49. Палитра read-only, управляется параметрами цветокоррекции и бюджета

- Сессия: пост-аудит 2
- Что решили: после квантизации палитра не редактируется напрямую. Нет операций «удалить цвет», «заменить цвет», «объединить цвета». Управление палитрой происходит через параметры, которые её порождают: 5 слайдеров цветокоррекции (яркость, контраст, насыщенность, температура, тени/света), бюджет цветов, кадрирование. - Статус: активно. Отменяет ручное редактирование палитры из решения №14.

### 51. Цветокоррекция — 5 слайдеров с дебаунсом 300 мс

- Почему: пять параметров покрывают большинство сценариев цветокоррекции бытового фото. Дебаунс предотвращает запуск тяжёлой квантизации на каждое микродвижение пальца.

### 52. Кадрирование — drag/pinch по 2D-развёртке всех целевых стен; зазоры переключаемо

- Сессия: пост-аудит 2
- Что решили: при назначении одного фото на несколько стен (панорама) — пользователь видит развёртку этих стен в одну линию и двигает фото поверх неё пальцами (один палец — drag, два — pinch для масштаба). Поведение фото в зазорах между стенами — переключатель: «сквозное» (по умолчанию, фото идёт через зазор как цельное полотно, кусок в зазоре теряется) и «с разрывом» (стены смыкаются вплотную для целей кадрирования, ничего не теряется).

- Альтернативы: только сквозное (отвергнуто — для плоских плиток теряется кусок без причины); только с разрывом (отвергнуто — для толстых плиток искажает реальный вид).
- Статус: активно

### 53. История undo/redo — снимки состояния без фотографий, лимит 50 шагов

- Сессия: пост-аудит 2
- Что решили: history-stack хранит снимки состояния всех настроек (комната, плитка, маски, палитра-параметры, кадрирование), но не сами фотографии. Замена фото в undo/redo не откатывается — пользователь загружает фото заново при необходимости. Лимит истории — 50 шагов.
- Почему: снимки проще и надёжнее дельт (нет риска расходящегося состояния при багах в обратных операциях). Фото отдельно — иначе история раздувается до сотен МБ. 50 шагов — комфортный запас, при отсутствии фото в снимках это копейки по памяти.
- Альтернативы: дельты с обратными операциями (отвергнуто — риск багов, избыточно); снимки с фото внутри (отвергнуто — раздутие памяти).
- Статус: активно. Уточняет решение №43.

### 54. Маски — частичная плитка считается целой (округление в большую сторону)

- Что решили: плитка вырезается из спецификации тогда и только тогда, когда она полностью находится внутри маски-препятствия. Частично попадающая плитка считается целой и идёт в спецификацию. Физическая обрезка по контуру маски — задача пользователя при укладке, не приложения.
- Почему: альтернатива (вырезать частичные плитки) оставляет пустое пространство по контуру маски — нелогично для всех сценариев (бумага, краска, плитка). Логичнее купить с запасом и обрезать на месте.
- Альтернативы: вырезать частичные плитки (отвергнуто); генерировать «дробные» плитки в спецификации (отвергнуто — неудобно для бумажной мозаики и краски).
- Статус: активно

### 55. Растяжка швов на краях стены отменена, остаётся только обрезка крайней плитки

- Сессия: пост-аудит 2

- Альтернативы: оставить растяжку (отвергнуто — конфликт с масками); центрирование с обрезкой с двух сторон (отложено — пользователь может реализовать сам, сдвинув начало сетки).

### 62. UI настройки углов — список в форме настроек комнаты

- Сессия: пост-аудит 2
- Что решили: переопределение направления перекрытия плиток на углах живёт в форме настроек комнаты, под списком стен. Каждый угол — отдельная строка с селектором: «автоматически» / «Стена N перекрывает» / «Стена M перекрывает». Рядом мини-схема результата (две полоски с разной толщиной).
- Почему: редкая операция, не нужна частая контекстная доступность. В 2D перекрытие не видно (стены показаны плоско), в 3D на мобильном палец не попадёт в узкий угол. Список в форме — единственное надёжное место.
- Альтернативы: тап в 2D-развёртке (отвергнуто — эффект не виден в 2D), тап в 3D (отвергнуто — узкая клик-зона на мобильном).
- Статус: активно. Закрывает Q7.

### 63. Замена фото — toast-уведомление о сбросе кадрирования

- Сессия: пост-аудит 2
- Что решили: при замене фото на стене новым с другими пропорциями — кадрирование автоматически сбрасывается, в углу всплывает toast «Кадр сброшен под новое фото», исчезает через 3 секунды. Без блокирующего диалога.
- Почему: замена фото — итеративная операция, диалог раздражает при повторных заменах. Toast информирует, но не блокирует.
- Альтернативы: блокирующий диалог (отвергнуто — раздражение), молчаливый сброс (отвергнуто — пользователь думает, что баг).
- Статус: активно. Закрывает Q12.

### 64. Формат файла проекта — поля format и version, валидация при импорте

- Сессия: пост-аудит 2
- Что решили: JSON-файл проекта на верхнем уровне содержит поля `format: "drape-project"` и `version: "1.0"`. При импорте — три уровня проверки: парсинг → сигнатура format → совпадение версии. При несовпадении версии — отказ. При битой структуре внутри — загрузка того, что можно, остальное игнорируется с предупреждением.
- Почему: без сигнатуры пользователь может загрузить любой JSON и получить непонятную ошибку. Версия нужна для будущих миграций при изменении формата.
- Альтернативы: без проверок (отвергнуто — плохой UX при ошибке).
- Статус: активно

### 65. Цветокоррекция — расширена до 5 слайдеров, обновление PROJECT.md

- Сессия: пост-аудит 2
- Что решили: PROJECT.md шаг 5 ключевого сценария обновлён с «яркость, контраст, насыщенность» до «яркость, контраст, насыщенность, температура, тени/света».

### 66. «Полезная ширина» разделена: сеточная ширина для расчёта плиток, маски не участвуют в распределении панорамы

- Сессия: финальная вычитка
- Что решили: переименовать «полезную ширину» в **«сеточную
  ширину»** (`grid_width`) — это длина стены минус толщина
  плиток соседей, перекрывающих в углах. Только этот параметр
  участвует в расчёте сетки плиток. Распределение фото-панорамы
  по стенам — пропорционально сеточной ширине, маски в
  распределении не учитываются. Фото лежит цельным полотном
  поверх развёртки, кусок над маской просто не отрисовывается
  (плитки в маске не кладутся).
- Почему: раньше один и тот же термин «полезная ширина» в
  ROOM_MODEL и PIXELIZATION значил разное (углы vs. маски), это
  ломало логику. Маски — механизм спецификации, не геометрии:
  они не должны искажать пропорции панорамы.
- Альтернативы: распределение пропорционально площади без
  масок (отвергнуто — фото сжимается на стене с большой маской,
  ломает цельность панорамы).
- Статус: активно. Уточняет решение №15.

### 67. Поле shape убрано из модели плитки в v1

- Сессия: финальная вычитка
- Что решили: поле `shape` в параметрах плитки удалено. В v1
  плитка всегда четырёхугольная (квадрат или прямоугольник, что

- Почему: зомби-поле, которое нигде не использовалось и
  усложняло модель. Other shapes — отдельная большая задача
  далеко за рамками v1.
- Альтернативы: оставить как задел на будущее (отвергнуто).
- Статус: активно

### 68. Нумерация стен — по порядку добавления, обход замкнут

- Сессия: финальная вычитка
- Что решили: стены нумеруются в порядке их добавления
  пользователем (Стена 1, Стена 2, …). Замкнутый контур
  обходится в том же порядке: после последней стены контур
  возвращается на первую. Угол с номером N — это угол между
  стенами N и N+1 (последний угол — между N и 1).
- Почему: правило «при равной толщине плиток меньший номер
  перекрывает» (№47) требует детерминированной нумерации.
  Порядок добавления — естественный и понятный пользователю.
- Альтернативы: нумерация по часовой стрелке от какой-то
  «первой» стены (отвергнуто — требует вычислить ориентацию).
- Статус: активно

### 69. Нумерация рядов плиток — снизу вверх, плиток в ряду — слева направо

- Сессия: финальная вычитка
- Что решили: ряды плиток на стене нумеруются от пола вверх
  (ряд 1 — нижний). Плитки в ряду — слева направо. Координаты
  плитки в режиме укладки: «Стена N, ряд K (от пола), плитка M
  слева».
- Почему: совпадает с физическим порядком укладки плитки и
  кирпича (с пола). Совпадает с математикой расчёта (ось Y от
  пола).
- Альтернативы: сверху вниз (отвергнуто — ломает аналогию с
  физической укладкой).
- Статус: активно

### 70. tile_overrides — все параметры плитки, кроме отсутствующих

- Сессия: финальная вычитка
- Что решили: на уровне стены через `tile_overrides` можно
  переопределить любой из параметров плитки: `tile_width`,
  `tile_height`, `tile_thickness`, `grout_width`, `grout_color`.
  Поле `shape` отсутствует в модели (см. №67).
- Почему: явно зафиксировать список, чтобы Claude Code не
  гадал. Все эти параметры независимы и могут осмысленно
  отличаться по стенам.
- Альтернативы: ограничить только толщиной (отвергнуто —
  негибко без понятной причины).
- Статус: активно. Уточняет решение №9.

### 71. Передача документации в Claude Code и порядок этапов

- Сессия: финальная вычитка
- Что решили: при запуске реализации в Claude Code передаются
  файлы из /planning/: PROJECT.md, ROOM_MODEL.md, PIXELIZATION.md,
  VIEWER_3D.md, EXPORT.md, TECH_STACK.md, DECISIONS.md.
  Не передаются: HANDOFF_PROTOCOL.md, PLAN.md, OPEN_QUESTIONS.md.
  Этапы реализации (из TECH_STACK §«Этапы реализации»)
  выполняются по одному, каждый — в отдельном чате Claude Code
  и в отдельной git-ветке с PR. Между этапами — handoff по
  тому же принципу, что в планировании.
- Почему: пользователь не программист, контролируемое
  движение этап-за-этапом снижает риск молча уйти не туда.
  DECISIONS даёт Claude Code контекст «почему так» при спорах.
- Альтернативы: один длинный чат на всю реализацию (отвергнуто
  — теряется фокус); этапы прямо в `main` без PR (отвергнуто —
  теряется видимая история и возможность откатить).
- Статус: активно
