# Пикселизация изображения

## Загрузка

- Форматы: JPEG, PNG, WebP
- Мягкий лимит: ~20 МБ. Предупреждение, не блокировка.
- Загрузка через стандартный `<input type="file">`

## Цветокоррекция (до пикселизации)

К исходному изображению применяются 5 глобальных слайдеров:

- **Яркость** (brightness)
- **Контраст** (contrast)
- **Насыщенность** (saturation)
- **Температура** (тёплый/холодный) — компенсация цвета лампы
- **Тени/света** (раздельные ползунки) — поднять тени без пересвета

При движении любого слайдера применяется **дебаунс 300 мс**: после
паузы запускается полный пересчёт (цветокоррекция → пикселизация →
квантизация → перекраска плиток). Один тяжёлый пересчёт вместо
сотен на каждое микродвижение пальца.

## Алгоритм пикселизации

**Метод:** area averaging (усреднение по области).
Цвет каждой плитки = средний цвет всех пикселей оригинала, попавших
в её область на исходном изображении (с учётом цветокоррекции).

Обработка выполняется в Web Worker + OffscreenCanvas (не блокирует
интерфейс). Для Safari старше 16.4 — фоллбек на обычный canvas
в главном потоке. Современные Chrome, Firefox, Edge, Yandex —
поддерживают OffscreenCanvas без проблем.

## Палитра — одна на проект

Квантизация (сведение всех уникальных цветов к ограниченному набору)
применяется **всегда** — без неё после area averaging получаются
тысячи уникальных цветов, что ломает спецификацию и легенду.

**Палитра — единая на весь проект.** Все стены квантуются вместе,
один набор цветов для всех. Это означает:
- В спецификации один «синий», а не четыре чуть отличающихся.
- Цельная панорама на нескольких стенах не разваливается на
  разные оттенки одного цвета.

**Параметры:**
- Бюджет цветов: 8–256, **дефолт 32**
- Алгоритм: медианное сечение (median cut)

**Сортировка палитры в UI:**
- По умолчанию — **по многочисленности** (от большего количества
  плиток к меньшему). Отражает реальный приоритет в работе:
  закупка, укладка.
- Переключатель — **по hue** (по цветовому кругу). Полезно для
  визуального восприятия композиции.

## Палитра read-only

После квантизации палитра **не редактируется напрямую**. Нет
операций «удалить цвет», «заменить цвет», «объединить цвета».

Управление палитрой происходит через параметры, которые её
порождают:
- 5 слайдеров цветокоррекции
- Бюджет цветов
- Кадрирование

Изменение любого из них перезапускает квантизацию (с дебаунсом).
Палитра обновляется автоматически.

Аргументация: фото — сбалансированная композиция. Ручное
вмешательство в палитру разрушает баланс. Резко упрощает
реализацию (нет undo для палитры, нет багов «удалили цвет →
заменили фото → куда переходят плитки»).

## Распределение по стенам

**Режимы назначения изображения (per-стена):**
- **Настраиваемая панорама** — одно фото на выбранные стены,
  растягивается пропорционально **сеточной ширине** каждой
  (см. ROOM_MODEL: сеточная ширина = длина стены минус толщина
  плиток соседей, перекрывающих в углах)
- **Своё фото** — каждой стене назначается отдельное изображение
- **Пустая стена** — без мозаики

Режимы комбинируются. В v1 — одна группа стен с панорамой.
Несколько групп («стены 1–2 одна, 3–4 другая») — в OPEN_QUESTIONS Q8.

Стены имеют имена по умолчанию (Стена 1, Стена 2, …),
пользователь может переименовать.

### Маски не участвуют в распределении панорамы

Когда фото растягивается на несколько стен, маски-препятствия
**не учитываются** при вычислении пропорций. Фото идёт цельным
полотном поверх развёртки сеточной ширины. Часть фото, попавшая
над маской, просто не отрисовывается — плитки в маске не
кладутся, и видеть «закрытый» фрагмент пользователю незачем.

Логика: маски — механизм спецификации, не геометрии. Учёт масок
в распределении сжимал бы фото на стене с большой маской и
ломал бы цельность панорамы. Подробное обоснование — DECISIONS №66.

## Кадрирование — drag/pinch по развёртке

При назначении одного фото на несколько стен пользователь видит
**развёртку этих стен в одну линию** (как полоска прямоугольников).
Сверху лежит загруженное фото — растянутое по полезной ширине всех
целевых стен пропорционально.

Управление пальцами:
- **Один палец** — сдвиг (drag) фото
- **Два пальца** — масштаб (pinch)

Видно сразу, какая часть фото попадёт на какую стену.

### Поведение фото в зазорах между стенами

Между стенами в развёртке есть зазоры с подписями (для удобства
ориентирования). При кадрировании поведение зазоров — **переключатель**:

- **Сквозное** (по умолчанию) — фото идёт через зазор как цельное
  полотно. Часть фото, попавшая в зазор, не используется (она в
  реальности теряется в углу комнаты из-за толщины плиток).
- **С разрывом** — для целей кадрирования стены смыкаются вплотную,
  ничего не теряется. Полезно, когда плитки плоские (нет физических
  потерь в углу) и хочется, чтобы фото шло без разрывов смысла.

### Если у каждой стены своё фото

Каждая стена кадрируется независимо. Тап на стену в развёртке —
работаешь с её фото отдельно.

### Замена фото на стене

При замене фото на новое с другими пропорциями кадрирование
автоматически сбрасывается (центрирование, fit под пропорции стены).
В углу всплывает **toast-уведомление** «Кадр сброшен под новое
фото», исчезает через 3 сек. Без блокирующего диалога.

При необходимости пользователь откатывает заменой через undo
(восстанавливаются настройки кадрирования, но не само фото — фото
вне истории).

## Обрезка и пропорции

При несовпадении пропорций изображения и стены:
- **По умолчанию:** обрезка (crop) с центрированием. Пользователь
  может сдвинуть кадр (через drag/pinch выше).
- **Альтернатива:** вписать целиком (fit) — пустые полосы по краям.

## Превью — два уровня

**Уровень 1 — плоское 2D-превью:**
- Стены развёрнуты в плоскость в одну горизонтальную линию
- Маски-препятствия видны как пустые прямоугольники в сетке
- Обновляется автоматически при изменении параметров (с дебаунсом)
- Здесь же отображается палитра в боттом-панели
- Обработка через Web Worker, интерфейс не зависает

**Уровень 2 — 3D:**
- Финальный просмотр (см. VIEWER_3D.md)
- Активируется явной кнопкой «Перейти к 3D»
- Возврат к настройкам — с предупреждением, что изменение параметров
  переразложит мозаику
